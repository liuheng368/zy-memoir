# Feature 阶段 2 — 架构设计

## 目标

在 **`plan.md`**（与 **`templates/plan.md.tpl`** 章节对齐）中充实**可评审**的技术方案，至少包含：**现状分析**、**页面交互细节**、**API 梳理**、**架构设计**、**技术选型 / 约束**、**`## 技术方案`**（文件改动清单、新增字段预估）、**`## 方案澄清事项`**（与 **`spec.md` `### 澄清事项`** 形态一致：**`#### Qn`** + **`- [ ]`…`⚠️` / `- [x]`…`✅`**）、**验收标准**（主流程 + 边界）、**`## 开发任务`**（层级勾选拆解）、**`## 实现计划`**。其中 **`## 实现计划`** 按**组**拆分任务；每组为**仓库相对路径 + 简述 + 测试 / 走查说明**的 `- [ ]` 列表，且与 **`android/ARCH.md`** 模块与依赖边界一致。

## 必读共享约定

- `.cursor/skills/feature/workflow-reference.md`：spec / **plan** 结构、组依赖、并行规则、**`kmp/` / `android/`** 路径与 **`ly.world.worldly…`** 包前缀；**勿**虚构本仓不存在的 `src/main/java/...`。
- **`android/ARCH.md`**：Android 壳与 KMP 依赖方向、Feature 模块与 `:app` 职责、跨 Feature 导航约定。
- **`android/DEV_GUIDE.md`**：KMP 数据模型 **`Kmp` 前缀**、网络、`expect/actual` 变更规范、新增模块 checklist。

## 设计要点

1. **依赖与分层**：遵守 **Android → KMP 单向依赖**；壳层 Activity/Screen 与 KMP 共享逻辑边界清晰；跨 Feature 用显式 Intent 等（见 ARCH）。**勿**在实现计划中假设本仓存在 Spring **Controller/Service** 式分层，除非任务明确在**独立后端仓库**（该仓路径写在 **`plan.md`**，仍须为真实路径）。
2. **任务粒度**：每个 checkbox 对应可独立验证的一项（单文件/小范围 PR、或一组紧密相关的 `build.gradle.kts`、Manifest、路由注册等）。
3. **组排序**：前置组全部 `- [x]` 后再开始下一组；同组内可并行。
4. **测试说明**：KMP 优先 **`commonTest` + `kotlin.test`**；Android 壳可用 **`test` / `testDebugUnitTest`**；仪器化仅必要时列入计划。避免「只写大而全 E2E、无单测落点」。

## `plan.md` 产出范围（阶段 2 须落盘，与 `plan.md.tpl` 对齐）

**禁止**仅写 **`## 实现计划`** 而其它章节仍为模板占位（**`-`**、空表、仅 HTML 注释无正文）。须按 feature 实质填写；**不适用**的条目显式写 **`N/A` 或「本期不涉及」** 并一句说明理由。

### 现状分析（`## 现状分析`）

须覆盖下列 **三个** `###` 小节（与 feature 无关则 **N/A** 并一句理由）：

- **`### 已有基础设施`**：网络层、路由、权限链、存储、已有 API/UseCase、与设计系统相关的底层能力等——列**本仓真实路径**或模块名，避免重复实现。
- **`### 已有可复用 i18n 字符串`**：`composeResources`、`strings.xml`、KMP `StringResource` 等**已有 key** 或可沿用文案的引用方式；新增文案须与实现计划勾连。
- **`### 已有可复用的 UI 组件`**：Compose/KMP 侧可复用 Screen、组件、主题扩展等——列**组件或文件路径**，与 **`## 架构设计`**（功能模块划分）、**`## 实现计划`** 承接。

### 页面交互细节（`## 页面交互细节`）

以 **`spec.md`** **`## 页面交互描述`**（含 **§1 页面结构**、多页串联）为**输入**，在 **`plan.md`** 做**实现向**细化；**单页 / 主 Screen** 可重复以下三个 **`###`**（多页则分 **Screen A / Screen B** 各写一组，或用小节标题区分）。与 feature 无关（无 UI）则整节 **N/A**。

- **`### 页面布局与素材引用（对齐 spec）`**：**不**以 plan 替代 spec 中的行为描述；须**互链** `spec.md` 对应小节标题或锚点，并列出 **`source/`** 下线框/截图的**仓库相对路径**（与 spec 嵌入图一致）。标明 plan 中模块编号与 **spec 图 / ASCII 分块**（①②③…）的**对应关系**。
- **`### 模块交互拆解`**：按模块（或按 spec 分块）逐条写清：**复用组件**、**i18n key**（若已定）、**可见性条件**（如某开关态下才展示子列表）、**手势 / 导航 / Bottom Sheet / 系统返回**、错空态；必要时用表格「模块 → 交互 → 备注」；与 **`## 验收标准`** AC 编号可交叉引用。
- **`### 页面涉及接口`**：本页（或本 Screen）范围内的 **`METHOD` + `path`**、**触发时机**（进页拉取、切换即提交等）；与上节**模块条目**逐条对应。**Response 与字段表**写在 **`## API 梳理`** 同名 path 下，本节可**互链**避免重复。跨页、鉴权、错误码策略等**全局**说明可简写并指向 **`## 架构设计`**。

### API 梳理（`## API 梳理`）

列出本次 feature **新用或变更**的 HTTP 接口（**无接口**则整节 **N/A**）。每个接口单独一个 **`###`** 小节（标题形如 **`### 4.1 查询设置 — 进入页面时调用`** 或 **`### API-1：…`**），且**至少**包含四块（与评审可读性对齐）：

1. **方法**：`GET` / `POST` / `PATCH` / `DELETE` 等（与网关/后端一致）。
2. **路径**：完整 path（如 **`/api/sns/v1/...`**），**勿**虚构未确认契约。
3. **Response Payload**：成功态 **JSON 示例**或 **Schema 摘要**（含嵌套/数组）；若有统一 **`data` 包裹**须写明；错误体可另起小节或指 **`## 架构设计`**）。
4. **字段映射**：四列表格 **`API 字段` | `类型` | `对应 UI` | `说明`**——`API 字段` 支持点分与 **`arr[].field`**；`对应 UI` 填 **spec/plan 模块编号**（如 **`③ 主态开关`**）或组件名；`说明` 写枚举取值、可见性条件、与 **`## 页面交互细节`** 的对应关系。

多条接口则复制 **`### API-n`** 结构；与 **`### 页面涉及接口`**、**`## 实现计划`** 中网络层任务**路径一致**。

### 架构设计（`## 架构设计`）

- **系统架构概览**：**Android 壳 ↔ KMP ↔ 网络/后端**（及 iOS 若适用）的数据与控制流边界；可附 **Mermaid** `flowchart` / `C4Context` 级简图或分段列表，且与 **`android/ARCH.md`**、**`android/DEV_GUIDE.md`** **不矛盾**。
- **功能模块划分**：写清 **`kmp/`、`android/`** 各模块职责、**依赖方向**、与 **`spec.md`** 用户路径的对应关系（**无**独立「范围与模块」表时，本节为工程模块清单真源）。

### 技术选型 / 约束（`### 技术选型 / 约束`，位于 `## 架构设计` 下）

须逐项覆盖（**与 feature 无关**则写 **N/A**）：

- **页面栈与导航**：Activity / Compose `NavHost` / 路由表、**返回栈**、与 **`spec.md`** 多页串联 / 系统返回 / 再入行为一致。
- **状态管理**：ViewModel、`StateFlow` / Compose `remember` 边界、跨页状态恢复、与 UI 真源。
- **分页 / 列表加载**：一次性拉全、分页、无限滚动、**虚拟列表**等——与 **`spec`** 及接口契约一致；无列表则 **N/A**。
- **其它**：网络（Ktor / OkHttp）、错误与重试、权限、`expect/actual`、缓存、并发等——按本 feature **择要**书写。

### 技术方案（`## 技术方案`）

实现级落地，与 **`## 架构设计`**、**`## 实现计划`** **一致**（**无**纯 UI 也无须改文件时两小节均可 **N/A**）。

- **`### 文件改动清单`**：四列表格 **`文件` | `改动类型` | `状态` | `说明`**——`文件` 列为 **`kmp/`、`android/`** 等**仓库相对路径**（可反引号包一层）；`改动类型` 用 **修改** / **新增**（必要时 **删除**）；`状态` 与开发进度对齐（**✅** / **⬜** / **进行中** 等，或链接 **实现计划** checkbox）；`说明` 一句写清路由常量、`when` 分支、ViewModel 职责、新增数据模型等（与参考图一致）。
- **`### 新增字段（预估）`**：用代码块（默认 **` ```kotlin `**，跨端可改用 Swift 等）列出 **UiState** / **ViewModel State** / **导航参数** 等**新增或变更**字段，含**类型**与**默认值**；用行内注释分组（如 **`// 主态`**、**`// 客态`**、**`// 好友列表`**）；与 **`## API 梳理`** 字段映射、**`## 页面交互细节`** 模块编号可对照。

### 方案澄清事项（`## 方案澄清事项`）

与 **`spec.md`** **`### 澄清事项`**（**需求 / 产品**向）**分工不同**：本节只收 **plan / 实现方案分叉**——动效与转场路径、导航栈与返回语义、状态拆分、技术选型 A/B、与 **AC** 强绑定的工程取舍等，供**研发或评审**勾选落定。

- **形态**（须与 **spec-creator**「澄清事项」一致）：每条 **`#### Qn：<一句话主题>`**，下列 **`- [ ]`…`⚠️`**（待选）或 **`- [x]`…`✅ 已确认`** / **`✅ 已确认`**；**勿**仅用三列表格代替 **Qn** 块。
- **写法示例**（与 spec 相同）：同一 **Qn** 下多条 **`- [x]`** 描述已定案路径（如「先停留往期页再进消费页」「返回与进入动效互为逆向」），句末 **`✅ 已确认`**。
- **互链**：可指向 **`spec.md`** **`## 页面交互描述`** 中已有行为描述，在本节收敛为**可执行**选项；**`## 开发任务`** / **`## 实现计划`** 中依赖某 **Qn** 的条目标 **「待 plan Qn」** 或互链锚点。
- **与前置校验关系**：**`spec.md` `### 澄清事项`** 未闭合仍按上文**禁止**落盘；**`plan.md` 本节**允许存在 **`- [ ]`…`⚠️`**（不阻塞阶段 2 落盘），但进入 **阶段 3（编码）**前，与**关键路径强相关**的 **Qn** 应已 **`- [x]`…`✅`** 或经 **TL / 用户书面放行**；否则须在 **`## 风险与未决事项`** 显式记录。
- **无方案分叉**：本节写 **无** 或 **N/A** 一句即可。

### 验收标准（`## 验收标准`）

- **主流程**：从 **`spec.md`** 抽取可测步骤，**`### AC-1`…`### AC-n`** 或模板中的 **输入 / 操作 / 预期** 结构，编号连贯。
- **边界 case**：权限拒绝、网络失败、空态/错态（若 `spec` 有定义）、重复提交、返回栈、deeplink（若支持）、多语言、老数据兼容等；与 **`spec.md`** **`### 澄清事项`** **已定案**勾选项一致，**禁止** AC 与 spec 冲突；与 **`plan.md` `## 方案澄清事项`** 中 **`- [x]`…`✅`** 已定方案**一致**，**禁止** AC 与 plan 方案冲突。

### 开发任务（`## 开发任务`）

面向**编码执行**的层级 **`- [ ]` / `- [x]`** 清单（**无**增量开发则整节 **N/A**）。

- **顶层**：按 **`kmp/`、`android/`** 下**文件**或 **Screen 聚合**（反引号路径 + 一句职责）；须与 **`## 技术方案` → `### 文件改动清单`** 行**一一对应**，避免「表里有文件、清单无条目」。
- **子项（缩进列表）**：按**实现顺序**或**页面区块**拆解——导航栏 / Hint / 主开关 / 选择器 / Bottom Sheet / 好友列表等；写明**复用组件**名（如 `SettingTopBar`、`WorldlySwitch`）、**状态**从哪来（ViewModel / `remember`）；复杂 Screen 可多级缩进。
- **标记**：依赖**设计稿、切图、产品文案、运营配置**且未闭环的条目，行首或行末标 **`⚠️`**；**API** 可单列顶层或子项「契约已确认 / 联调通过」，与 **`## API 梳理`** 对齐。
- **与 `## 实现计划` 分工**：**开发任务** = **叙事 + UI 细拆 + 文件内步骤**；**实现计划** = **按仓库路径分组 + 测试 / 走查**；二者条目应**可交叉映射**，避免仅写实现计划而无开发任务（或反之）。

### 实现计划（`## 实现计划`）

见下文 **「执行步骤」** 与 **「设计要点」**；路径须为 **`kmp/`、`android/`** 等本仓真实前缀。

## 前置校验（须先于「执行步骤」）

在 **`plan.md`** 编写或扩充技术方案正文（自 **`## 概述`** / **`## 现状分析`** / **`## 页面交互细节`** / **`## API 梳理`** / **`## 架构设计`** / **`## 技术方案`** / **`## 方案澄清事项`** / **`## 验收标准`** / **`## 开发任务`** 起，直至 **`## 实现计划`**）之前，**必须先**校验 **`spec.md`** 中 **`### 澄清事项`**（及 **`### 澄清项`**、**`### 二、澄清事项`** 等**等价**小节，见 **`spec-creator/SKILL.md`**）是否已**全部确认**。

1. **Read** 目标 **`docs/features/<slug>/spec.md`**，定位澄清小节（自 **`### 澄清事项`** 起至下一 **`## `** 级标题之前）。
2. **未决判定**（命中任一即**不得**继续落盘实现计划）：
   - **`#### Qn`** 下存在 **`- [ ]`** 且行末含 **`⚠️`**，或存在明显「待确认 / TBD」而无 **`✅` / `✅ 已确认`** 的勾选项；
   - **旧版**仅三列表格、无 **`#### Qn`**：表中「建议 / 默认假设」列大量空白，或「待产品确认」类表述未闭环。
3. **若存在未决**：**禁止**在本轮 **`Write`/`StrReplace`** 写入 **`plan.md`** 的 **`## 现状分析`**、**`## 页面交互细节`**、**`## API 梳理`**、**`## 架构设计`**、**`## 技术方案`**、**`## 方案澄清事项`**、**`## 验收标准`**、**`## 开发任务`**、**`## 实现计划`** 等阶段 2 正文（**禁止**为赶进度编造默认澄清）；对话中**逐条列出**未决 **Qn** 或表格行及原文，建议用户 **`/assist spec`** 或手工更新 **`spec.md`** 后再执行 **`/assist plan`**。（**说明**：**`spec.md`** 澄清未决时**整表禁止**；**`plan.md` `## 方案澄清事项`** 内 **`- [ ]`…`⚠️`** 为**方案向**待定，**不**替代本条对 **spec** 的校验。）
4. **若无澄清小节**：须与用户确认「本 feature 无澄清项」或提示先 **`/assist spec`** 补 **`### 澄清事项`**；**不得**静默跳过。
5. **仅当**澄清已闭合（各 **`- [x]`…`✅`**，或等价已定案表述）或用户**显式**声明「接受当前未决项、仍要写实现计划」后，方可进入下方 **「执行步骤」**。

## 执行步骤

1. **（前置已通过）** **Read** **`spec.md`**（**`## 需求描述` / `## 问题定义`**、**`## 页面交互描述`**）及既有 **`plan.md`**；若 **`plan.md`** 的 **`## 验收标准`** 仍占位，则从 **`spec`** 与已定案 **澄清** 推导 AC 草案。
2. 按 **「`plan.md` 产出范围」** 依次 **Write / StrReplace** 充实：
   - **`## 概述`**；
   - **`## 现状分析`**：**已有基础设施**、**已有可复用 i18n 字符串**、**已有可复用的 UI 组件**（见上节；须列本仓可引用路径或 **N/A**）；
   - **`## 页面交互细节`**：**页面布局与素材引用**、**模块交互拆解**、**页面涉及接口**（见上节；须 **Read** **`spec.md`** 页面结构并与 **`source/`** 路径对齐；无 UI 则整节 **N/A**）；
   - **`## API 梳理`**：逐条接口写清 **方法**、**路径**、**Response Payload**、**字段映射**表（见上节；无 HTTP 则 **N/A**）；
   - **`## 架构设计`**：**系统架构概览** + **功能模块划分**（见上节）；
