# Feature 阶段 4 — 代码审查

## 目标

确认实现满足 **`plan.md`**（默认）或旧版 **`spec.md`** 的 **`## 验收标准`**，并符合 **`android/ARCH.md`** 与 **`android/DEV_GUIDE.md`** 工程约定；**汇总问题严重级别**，推动修复后再进入阶段 5。

**必须与聊天输出同步**：把审查结果**落盘到当前 Feature 的 `plan.md`**（见下节「写回 plan」）。**`/assist feat`（`feature/SKILL.md`）阶段 4→5 只读载体文件，不读对话历史**；仅口头结论会导致 **plan 不更新 → 永远阶段 4**。

## 必读

- `.cursor/skills/feature/workflow-reference.md`（spec / plan 结构、分支命名、checkbox 规则）
- `docs/features/<feature-name>/spec.md`（需求与分支；旧版可仍含 AC）
- `docs/features/<feature-name>/plan.md`（**默认含 AC**、实现计划；**代码审查**写回本文件）
- `android/ARCH.md`（架构与分层约定）
- `android/DEV_GUIDE.md`（开发与协作约定）

## 检查清单（建议）

对照 `android/ARCH.md`、`android/DEV_GUIDE.md` 核对，并关注：

| 类别 | 要点 |
|------|------|
| 需求对齐 | 每条 AC 有对应实现或测试覆盖说明 |
| 架构与依赖 | Android→KMP 单向依赖、Feature/KMP 模块边界、跨 Feature 导航与 `:app` 壳职责符合 ARCH |
| 数据与规范 | 新增 KMP 数据模型、命名与协作约定符合 DEV_GUIDE（如 `Kmp` 前缀等） |
| 安全与日志 | 敏感信息不记入日志；异常与用户可见错误处理合理 |
| 测试 | 关键逻辑有单测或覆盖说明；非仅有慢速全上下文测试 |
| 配置 | 新增配置有说明；不提交密钥 |
| 范围 | 无不相关重构或顺带大改（除非 spec 单列） |

## 严重级别

- **HIGH**：安全、资损、破坏兼容性且无迁移说明、AC 未满足、明显逻辑错误、无测试且无理由。
- **MEDIUM**：可维护性、边界情况、日志与监控不足。
- **LOW**：风格、命名、注释。

## 写回 `plan.md`（必做，避免阶段 4→5 断裂）

审查结束前，Agent **必须**在当前 **`docs/features/<slug>/plan.md`** 中维护 **`## 代码审查`** 小节（与 **`.cursor/skills/feature/workflow-reference.md`** 模板一致），**不得**只做聊天回复不写文件。**若目录无 `plan.md` 或用户明确维护旧版**：可暂写 **`spec.md`** 同节（与 **feature/SKILL**「兼容」检测一致）。

1. **定位**：若已有 **`## 代码审查`**，用 **StrReplace** 更新其下 **`结论` / `严重问题` / `备注`** 三行；若无该节，在 **`## 实现计划`** 之后（或 **`plan.md`** 全文末尾）**插入**该节。
2. **结论**（三选一，与阶段路由可读性对齐，**建议字面使用下列词之一**）：
   - **`通过`** — 无未关闭 **HIGH**（及团队若仍跟踪的 **CRITICAL**），可进入阶段 5；**「已通过」** 与 **「通过」** 同义，但**优先写「通过」**以减少歧义。
   - **`需修改`** — 存在未修复或未书面豁免的 **HIGH**。
   - **`待审查`** — 尚未完成本次审查（一般不应作为本轮收尾；仅当审查中断、留待下次继续时使用）。
3. **严重问题**：列出本轮仍开放的 **HIGH**（可多条）；无则写 **「无」**。**MEDIUM/LOW** 可写入 **备注** 或对话摘要，不必挤在「严重问题」行，但若 HIGH 已关闭、仅余 MEDIUM，**结论**仍可用 **「通过」** 并在 **备注** 中说明遗留 MEDIUM 与用户是否接受。
4. **与对话一致**：聊天里的「摘要 / 问题列表」与 **plan 内三行** 应**同一结论**，避免 plan 写通过、聊天仍列 HIGH 等矛盾。

## 输出格式（对话 + 落盘）

1. **摘要**：是否建议进入阶段 5（与 **plan** **结论** 一致）。
2. **问题列表**：分级列出，含文件路径与修改建议。
3. 存在 **HIGH** 时：**不得**进入阶段 5；**plan** **结论** 写 **`需修改`**，**严重问题** 写明 HIGH 项；先修复并重新审查或记录豁免（需用户确认）。
4. **落盘**：本轮结束前完成对 **`plan.md`「## 代码审查」** 的 **StrReplace/Write**（见上节）。

## 下一步

仅剩 **LOW** 或用户书面接受 **MEDIUM** 遗留项、且 **plan「代码审查」结论已为「通过」**（或同义「已通过」且团队认可）时，提示用户执行 **`/assist done`** 进入阶段 5（提交与收尾）。
