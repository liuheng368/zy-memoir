# Feature 开发流程路由

Feature 开发按 **阶段 0（初始化）** 与 **阶段 1～5** 顺序推进；具体约定见 **`.cursor/skills/feature/workflow-reference.md`**（分支与 slug 映射、spec 结构、多栈路径示例）。

| 阶段 | Skill 路径 | 说明 | 核心产出 |
|------|------------|------|----------|
| **0** | `.cursor/skills/spec-creator/SKILL.md` | 尚无 `spec.md`：建目录与模板 | 提示执行 **`/assist spec`**（Token **`spec`**）；产出 `docs/features/<slug>/` 骨架与初始 `spec.md` |
| 1 | `.cursor/skills/feature-requirements/SKILL.md` | 需求澄清 | spec.md（问题/需求、分支名）+ **`plan.md`**（**`## 验收标准`** / AC，默认） |
| 2 | `.cursor/skills/feature-plan/SKILL.md` | 架构与实现计划 | **`plan.md`**「实现计划」与 checkbox |
| 3 | `.cursor/skills/feature-code/SKILL.md` | 实现与测试 | 代码 + 测试；勾选 **`plan.md`** 任务 |
| 4 | `.cursor/skills/feature-review/SKILL.md` | 代码审查 | **`plan.md`**「代码审查」；处理 HIGH |
| 5 | `.cursor/skills/feature-commit/SKILL.md` | 提交与收尾 | commit、推送、MR/PR 说明 |

## Feature 标识（slug）解析

1. 用户可显式给出 **slug**（kebab-case，与 `docs/features/<slug>/` 一致）。
2. 若未指定且当前分支为 **`feat/<rest>`**：取 `rest`，按 `workflow-reference.md` **分支名与目录名**规则生成候选路径并依次检查 **`docs/features/<候选>/spec.md`** 是否存在：**`-` 与 `_` 视为等价分隔符**（两种形态都试）；**其余字符与目录名严格一致**（大小写、拼写不自动纠正）。
3. 若分支为 **`fix/*`**、**`main`**、**`develop`** 或其它非 `feat/*` 形式，**无法自动推断** slug → 须请用户给出 slug 或 `spec.md` 的完整相对路径后再检测。

## 当前阶段检测

对解析到的 **`docs/features/<slug>/spec.md`** 及同目录 **`plan.md`**（须先完成 **Feature 标识（slug）解析**；若无法得到 slug，请先请用户给出 slug / `spec.md` 路径或切换到 `feat/<name>` 分支，**不**套用下表阶段号）。**验收标准（AC）** 的**验收载体**与 §1「**验收载体（AC）**」同规则。**实现计划 / 代码审查**的**载体**判定见 **`workflow-reference.md`**：**`plan.md`** 中对应节**已有实质 checkbox** 时优先 **`plan.md`**；若仅有空节（模板占位、无 `- [` 行）则**降级**看 **`spec.md`**（兼容无 **`plan.md`** 或旧文档）。

### 0. spec 是否存在

- 若 slug 已解析，且任一候选路径下**均无** `spec.md` → **阶段 0**：明确提示用户在仓库根执行 **`/assist spec`**（等同 Token **`spec`**，读取 **spec-creator** Skill）；可选 `--slug` / `--branch` 与 `workflow-reference.md` 分支规则对齐。若用户已在对话中声明即将自行建目录，仍应给出 **`spec`**（spec-creator）路径以便一致执行。

### 1. 需求与验收是否就绪

**验收载体（AC）**：**`plan.md`** 若存在 **`## 验收标准`**、至少一条 **`### AC-`**，且**非**「仅模板占位」（即至少一条 AC 的「输入 / 操作 / 预期」之一已有实质文字，非空且非单独 **`...`**）→ 在 **`plan.md`** 上检测 AC；**否则若** **`spec.md`** 含 **`## 验收标准`** 且至少一条 **`### AC-`** → 在 **`spec.md`** 上检测（**兼容**：新建骨架里 **`plan.md`** 仍为占位 AC、实质 AC 暂在 **`spec.md`** 的旧文档）；**否则**在**先存在者**上检测（通常 **`plan.md`** 模板已带空 AC 节）。

以下**不满足**则 → **阶段 1**：

- **需求**（均在 **`spec.md`**）：无「问题定义」且无实质「需求描述」（仍为 spec-creator 占位句、或背景为空）；
- **验收**（在**验收载体**上）：无「验收标准」章节，或 AC 仍为空白模板；
- **分支**（通常在 **`spec.md`**）：全文无 `` `feat/...` `` 或独立「分支名称」小节。

（「问题定义」与「需求描述」二有一即可，见 `workflow-reference.md` 等价表。）

### 2. 实现计划（载体：**`plan.md` 有实质 checkbox 则优先**，否则 **`spec.md`**）

1. **先定载体**：**若** **`plan.md`** 存在、含 **`## 实现计划`**、且该节内至少有一条 **`- [`** 开头的 checkbox 行 → 载体为 **`plan.md`**；**否则若** **`spec.md`** 含 **`## 实现计划`** 且至少一条 checkbox → 载体为 **`spec.md`**；**若两者均无有效 checkbox** → **阶段 2**（未就绪）。
2. **再在载体上检测**：
   - 无「实现计划」章节，或无任一 `- [ ]` / `- [x]` → **阶段 2**
   - 仅含**空 checkbox**（`- [ ]` 后无实质文字，如 `- [ ] ` 或 `- [ ]  `）也视为未就绪 → **阶段 2**

### 回溯校验（阶段 3+ 硬关卡）

> **顺序短路问题**：阶段 0→2 只检查结构标记（spec 是否存在、有无 checkbox），一旦结构齐全即放行。如果 AC 或需求描述在阶段 1 检测后被清空、误删、或从未被实质编辑（仅保留 spec-creator 占位内容），后续阶段不会再检查，空 spec 可直接进入代码审查。

当检测**已通过阶段 2**（在载体文件上实现计划存在且含 checkbox）、即将进入阶段 3 或更高时，**必须**回溯重检阶段 1 的两项内容条件：

1. **需求**：在 **`spec.md`** 上，无「问题定义」且无实质「需求描述」（仍为 spec-creator 占位句、或背景为空）；
2. **验收**：在**验收载体**（见 §1）上，无「验收标准」章节，或 AC 仍为空白模板（如 `### AC-1：<标题>` 中「输入 / 操作 / 预期」均为 `...` 或空白）。

**任一不通过 → 强制判定为阶段 1**，终止后续检测，并在输出中说明：

- 阶段号：**1**
- 阻塞原因：`需求描述或验收标准（AC）为空 / 仍为占位模板。实现计划已存在但 AC 缺失，须先补齐 AC 再继续。`
- 建议动作：进入 `feature-requirements`（阶段 1 Skill）补写需求与 AC。

### 3. 实现任务是否完成（与 §2 同一载体）

- 实现计划中仍存在 `- [ ]` → **阶段 3**
- **全部**为 `- [x]` → **阶段 4**（进入审查）

### 4. 审查与收尾（载体：**`plan.md` 存在且含「代码审查」则优先**，否则 **`spec.md`**）

1. **先定载体**：**若** **`plan.md`** 存在且含 **`## 代码审查`** → 载体为 **`plan.md`**；**否则若** **`spec.md`** 含 **`## 代码审查`** → 载体为 **`spec.md`**；**若均无** → **阶段 4**。若两者**同时**存在该节，以 **`plan.md`** 为准，并应删除 **`spec.md`** 内重复节以免双真源。
2. **再在载体上检测**：

- 无「代码审查」小节，或结论为「待审查」「需修改」，或 **HIGH** 未关闭（且无豁免）→ **阶段 4**
- 「代码审查」结论为 **通过**（或同义 **「已通过」**），且严重问题已处理或豁免 → **阶段 5**（结论须来自 **载体文件落盘**，见 **`feature-review`** Skill；仅聊天无更新仍视为未完成审查 → **阶段 4**）

## 输出

1. 当前使用的 **slug** 与 **spec / plan 路径**（阶段 0 时可写「候选路径均不存在 `spec.md`」）；若已判定载体，注明 **AC**、实现计划与代码审查各自在 **`plan.md`** 还是 **`spec.md`（兼容）**。
2. **阶段（0～5）**与已完成 / 未完成任务概览。
3. 下一步应打开的 **Skill 文件路径**（上表之一；**阶段 0** 为 **spec-creator**，**非** `feature-*` 前缀）。
4. 阻塞项（如无法解析 slug、无 spec、路径候选均不存在、前置组未完成、**回溯校验未通过**）。
