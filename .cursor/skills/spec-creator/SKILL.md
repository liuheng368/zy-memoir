# Spec Creator（需求准备）

在 **`docs/features/<slug>/`** 下维护**两份主文档**及 **`source/`**、**`prd/`** 子目录（与脚本、模板一致），**职责边界**如下：

| 文件 | 职责 |
|------|------|
| **`spec.md`** | **需求的详细阐述**：背景、范围、**`## 页面交互描述`**（**§1 页面结构**：**优先** **`./source/`** 内页面截图嵌入，**否则** **ASCII 示意结构图**；区域交互 / **操作与动效及跳转** / 单页与弹层串联 / **多页面串联**（**用户路径**；**勿**写「涉及模块」工程清单）；**无接口**）、澄清、分支名；**不**承载**涉及模块 / 改动矩阵**、组件复用表、Design Token、实现级技术方案；**不写** **`## 实现计划`** / **`## 代码审查`**（默认在 **`plan.md`**）。**`## 验收标准`（AC）** 默认在 **`plan.md`**（兼容旧文档仍写在 **`spec.md`**）。 |
| **`plan.md`** | **技术方案全文**（文档级）：**`## 概述`**、**`## 现状分析`**（已有基础设施、可复用 i18n、可复用 UI 组件）、**`## 页面交互细节`**（对齐 **`spec.md`**：布局引用、`source/` 路径、模块交互拆解、**本页** `METHOD`+`path`）、**`## API 梳理`**（逐条接口：**方法**、**路径**、**Response Payload**、**字段映射**表；与页面模块编号互链）、**`## 架构设计`**（含**功能模块划分**（**`kmp/`、`android/`** 职责与依赖）与 **`### 技术选型 / 约束`**；全链路数据流 / 跨页契约 / UI·Token 等并入此节或 **`## 实现计划`** 条目，**无**独立「接口与调用顺序」等章）、**`## 技术方案`**（**文件改动清单**、**新增字段（预估）**）、**`## 方案澄清事项`**（**`#### Qn`** + **`- [ ]`…`⚠️` / `- [x]`…`✅`**；**方案向**，与 **`spec.md` `### 澄清事项`** 分工）、**`## 风险与未决事项`**、**`## 验收标准`（AC）**、**`## 开发任务`**（层级 **`- [ ]`**：按文件/Screen 拆解）、**`## 实现计划`**（checkbox）、**`## 代码审查`**。**以 spec 需求为输入**撰写 AC 与技术方案。 |
| **`source/`** | **研发辅助素材**：线框、**页面截图**（**`spec.md`** §1 嵌入）、流程图、抓包、会议纪要等；**非** PRD 原文仓。模板 **`templates/source/README.md`**。 |
| **`prd/`** | **上游产品文档留存**：PRD、产品方案 PDF/DOCX/PPT、评审基线等；**大文件**建议 **Git LFS**（随仓库 **`.gitattributes`**）。与 **`source/`** 分工见 **`templates/prd/README.md`**。 |

与 **`.cursor/skills/feature/workflow-reference.md`** 对齐；**五阶段路由仍以 `spec.md` 为唯一入口**。脚本 **A** 与推荐 **B** 会一并初始化 **`source/README.md`** 与 **`prd/README.md`**（模板 **`_shared/create-feature-spec/templates/source/`**、**`templates/prd/`**）。

## Assist `spec` — 模式 ③：`prd/` / `source/` 文件整理（**优先判定**）

在判定 **① / ②** 之前，若命中本节 **触发条件**，**不再**按 ① / ② 默认流程执行；须 **Read** **`.cursor/skills/spec-creator/feature-doc-tidy.md` 全文** 并逐条落实。本模式允许在 **`docs/features/<slug>/`** 内 **移动 / 重命名** **`prd/`**、**`source/`** 文件，并对 **`spec.md`**、**`plan.md`**、**`prd/prd.md`** 等做 **StrReplace** 以修正 **Markdown 图片与链接**（为修正引用而改 **`spec.md`** 不违反「Assist 入口」意图）。

### 触发条件（**白名单**，须**同时**满足）

1. **消息**以 **`/assist spec`**（`/Assist SPEC` 等同）开头或含该 Token 段；去掉 **`/assist`**、**`spec`** 及**所有** **`@…`**（从 **`@`** 起至下一空白或行尾的路径，含 **`@docs/...`**）后，对剩余串 **trim**。
2. **剩余串**须与下列之一 **完全一致**（不多字、不少字、无前后标点或其它中文）：**`整理`**、**`整理文件`**、**`文件整理`**、**`整理一下`**。
3. **示例（命中③）**：`/assist spec 整理`、`/assist spec 整理文件`、`/assist spec 文件整理`、`/assist spec 整理一下`；以及 **`@docs/features/foo/spec.md` 整理**（去掉 `@…` 后为 **`整理`**）。

**不命中③（走 ①/②）**：如 **`/assist spec 整理 prd`**、**`整理一下需求`**、**`tidy source`**、仅 **`整理`** 但**未**发 **`/assist spec`** 等——**一律**不按③执行。

### 执行后对用户回复（须简短）

- 列出 **旧路径 → 新路径 / 已删除** 要点；若有**是否整页**无法判定之图，列文件名与体积请用户确认**保留或删除**（见 **`feature-doc-tidy.md`** B.2）。
- **`prd/`** 整理后须已按 **`feature-doc-tidy.md`** A.2 **第 5 步** 核对 **`prd.md`** ↔ **`prd/imgs/`**；若仍有存疑链接，在回复中单独列出。
- 提醒：后续需求迭代仍用 **`/assist spec` + 材料**（模式 ②）；**AC** 在 **`plan.md`** 时须 **`@plan.md`**。

### 裸 `/assist spec`（无附加需求正文、且已排除模式③）

**定义（裸触发）**：去掉 **`/assist`**、**`spec`** 及**所有** **`@…`** 路径片段后 **trim**，**无**需求/编辑类正文，或**仅有寒暄**（如「你好」「谢谢」）；且**不满足**下方「模式 ② 任一条件」。

**定位例外（仍属裸触发，但不自动写盘）**：整条消息在去掉 Token 与空白后，**仅有**一个 **`@…/spec.md`**（或单个 `@` 指向某 **`spec.md`**）**且无任何其它字符** → 视为**只打开/定位文件**，**不得**编辑 **`spec.md`** / **`plan.md`**。

**行为分支**（须先 **Glob** 目标 **`docs/features/<slug>/spec.md`**）：

1. **`spec.md` 不存在**（或目标目录尚无可作为阶段入口的 **`spec.md`**）：**初始化** — 按「落地方式 **A/B**」创建 **`spec.md` + `plan.md` + `source/README.md` + `prd/README.md`**（目录不存在用 **A**；目录已存在缺文件用 **B**）。
2. **`spec.md` 已存在**：
   - **验证 PRD**：优先 **`prd/prd.md`**；若无，再查 **`prd/`** 下是否有其它可作**主产品稿**的 **`.md`**（**排除**仅 **`README.md`**）；仍无 → **不编造**业务需求；简短回复：须补 PRD（可先 **`/assist spec 整理`**）或发 **`/assist spec` + 需求描述**。
   - **若主 PRD 存在**：**Read** 该 **`prd/*.md`**、现有 **`spec.md`**，并 **List/Glob** **`source/pages/`**（整页图）、**`prd/imgs/`**（与 PRD 插图一致即可）；**本轮须落盘 `spec.md`**：按 **`spec.md.tpl`** 结构从 PRD **提炼** **`## 需求描述`**、**`## 页面交互描述`**（§1 **优先**嵌入 **`./source/pages/`** 已有图；不足则 **ASCII**）、**`## 关联文档`** 链 **`./prd/prd.md`**（若主文件不叫 `prd.md` 则链实际相对路径）、**`### 澄清事项`** 中列 PRD 缺口（**须**用 **`#### Qn：…`** + **`- [ ]`…`⚠️`** 形态，见「澄清事项」专节）；**遵守「Assist 入口」**：未 **`@plan.md`**、未点名 **`plan.md`** 时**不**擅自大改 **`plan.md`**（可保留占位或一句「技术节待 `@plan.md`」）。

**`/assist spec` + 需求描述（或跟车材料）**：命中 **模式 ②**，**不适用**本节「裸触发」分支；按下方「模式 ②」「章节映射」**既有逻辑**执行（Read → StrReplace/Write）。

### 模式 ① / ② 判定硬规则（避免 Agent 误判）

**判定顺序**：**先**按上文 **「模式 ③：文件整理」** 检测；**已命中③则不再使用**下列 **① / ②** 规则。

**再**去掉 **`/assist`**、**`spec`**，再 trim；**忽略**仅用于 @ 文件的 Cursor 元数据噪声（若有），以**用户可见中文/英文意图**为准。

**模式 ② — 满足以下任一即判为 ②**（**不得**判为 ①）：

1. **跟车实质内容**：需求/架构/接口**段落或列表**、粘贴文档、明确「把某节改成…」「增加 AC-3」等。
2. **编辑类意图**（哪怕**无具体段落**）：出现指向文档的**修改、优化、完善、补充、更新、修订、润色、扩写、精简、重写、合并章节、加验收标准（通常须 **`@plan.md`**）、对齐架构/接口**等含义的表达（含常见英文：`update` / `refine` / `polish` / `expand` / `fix` **且语境为 spec**）。
3. **`@spec.md` 且有任何附言**（哪怕一个字，如「改标题」「补 AC」）。

**模式 ①（= 裸触发路径）— 满足以下之一**，**且**无 **②** 中任一触发：

1. 去掉 **`/assist`**、**`spec`** 与 **`@…`** 后 **trim**，正文**为空**或**仅有寒暄** → 按上文 **「裸 `/assist spec`」** 分 **`spec.md` 有无** 与 **PRD 有无** 执行（**非**「仅 `@…spec.md` 零附言」）。
2. **仅** `@…/spec.md`（或单个 `@` 指向 **`spec.md`**）**且无任何其它字符** → **定位例外**：**不编辑**（与「裸触发」写 **`spec.md`** 分支**互斥**）。

**模糊短句**（如「优化一下」「完善 spec」「改好点」）：

- **一律判为 ②**（属于「编辑类意图」），**禁止**因「没有长材料」退回 ①。
- **`spec.md` 已存在**：**Read** 全文 → **不得**无依据重写；若用户未指定改哪里：
  - 列出 **2～5 条**可改进点（占位句、缺 **AC**、缺 **架构/接口** 节、与 **workflow-reference** 不一致等），**追问**优先改哪一节；**或**建议用户 **`/assist specre`** 做规范审查后再来 **`/assist spec`** 带明确指令落盘。
- **`spec.md` 不存在**：先 **A/B** 建骨架，再同上**列改进方向**或请用户补充材料；**不**编造业务需求填满全文。

**与「模式 ②：章节映射」的关系**：硬规则只解决 **① vs ②**；进入 ② 后，**有材料**按章节映射写入；**材料不足**按上条「模糊短句」处理。

**解析目标路径**（两模式共用）：与 **feature** Skill 相同——优先用户 **`@` 指向的文件或目录**：若 **`@` 为 `docs/features/<slug>/` 目录**（或未写具体文件名），**主目标**为该目录 **`spec.md`**（阶段路由入口）；若用户 **`@plan.md`** 则以 **`plan.md`** 为 Write 目标。若 **`@` 具体到 `.../spec.md`** 则用之；否则当前分支 **`feat/<rest>`** → 候选 **`docs/features/<rest>/spec.md`**（`-` / `_` 互换）；无法解析 slug 时**先请用户**给出路径或 slug。

**Assist 入口与可编辑文件（硬约束）**

- 用户仅发 **`/assist spec`（Token `spec`）+ 材料**，且**未** **`@plan.md`**、材料中也**未点名**要改 **`plan.md`** 时：**本回合只读写的目标文件为 `spec.md`**（可把技术/UI 要点**摘要**进 `spec.md` 或写「待补：见 `plan.md`」——**不**擅自修改 **`plan.md`**）。**该命令默认只驱动 `spec.md` 变化**。**`plan.md`** 内 **`## 验收标准`（AC）** 的增删改须 **`@plan.md`** 或同条消息**明示**改 **`plan.md`**；**例外**：**`spec.md`** 已存在旧版 **`## 验收标准`** 且用户未提供 **`plan.md`** 目标时，可继续在该节迭代并**建议**迁移至 **`plan.md`**。
- 须落盘 **`plan.md`** 时：用户应 **`@docs/features/<slug>/plan.md`**，或在同一条消息里**写清**「请更新 **`plan.md`** …（含 AC / UI/Token 节）」；Agent **再** Read/Write **`plan.md`**。

**目录约定**：**`spec.md`** 与 **`plan.md`** 同目录；模式 ① 脚本 **A** 会一次性创建 **`spec.md` + `plan.md` + `source/README.md` + `prd/README.md`**。缺 **`plan.md`** / **`prd/`** / **`source/`** 的旧目录可用方式 **B** 从 **`.cursor/skills/_shared/create-feature-spec/templates/`** 下对应 **`*.tpl`** / **`source/README.md`** / **`prd/README.md`** 补写。

### `/assist spec` 与多轮对话（须支持）

- **允许**：同一 **`docs/features/<slug>/`** 下，用户可**连续多轮**发 **`/assist spec` + 补充说明**，用于**确认需求**、**收敛方案边界**、**补充边界 case**；**不因上一轮已写过而拒绝继续改**；每轮仍按上文 **模式 ① / ②** 与 **「Assist 入口」** 判定是否落盘、改哪些文件。
- **推荐节奏**：
  1. **需求与澄清**：多轮只带 **`/assist spec`**（未 **`@plan.md`**、未点名 **`plan.md`**）时，**只迭代 `spec.md`**——`## 需求描述`、**`## 页面交互描述`**（五节；**§1** **先扫 `source/` 截图，有则嵌入，无则须含 ASCII 示意结构图**（纵向复杂页）；须含**多页串联、动效与页面跳转**等行为级描述；**禁止**在 **`spec.md`** 写 **「涉及模块」** 专节或工程改动矩阵——**`kmp/`**、**`android/`** 模块与路径写在 **`plan.md`** **`## 架构设计`**（功能模块划分）或 **`## 实现计划`**，**`## 页面交互细节`**、**`## API 梳理`**、**`## 技术方案`**、**`## 方案澄清事项`**、**`## 开发任务`**（工程向细化须 **`@plan.md`** 或同轮明示）等节）、**`### 澄清事项`**（**默认** **`#### Qn：…`** 下 **`- [ ]`** / **`- [x]`** 任务列表，见专节；须纳入**文档缺口、需求冲突、待确认边界**；与 **`### 澄清项`**、**`### 二、澄清事项`** 视为同一节）；技术/实现细节可先写「**待确认：见下轮**」或「**待补：`plan.md`**」占位。**AC** 默认在 **`plan.md`**：须 **`@plan.md`** 或同条消息明示改 **`plan.md`** 时在本轮落盘 **`## 验收标准`**；仅 **`/assist spec`** 时可在澄清中记「待写 AC」并提示下一轮回 **`@plan.md`**。
  2. **验收与技术方案**：用 **`@docs/features/<slug>/plan.md`** 或正文**明确写**「请更新 **`plan.md`** …」维护 **`## 验收标准`（AC）**、**`## 页面交互细节`**（对齐 spec：布局引用、模块交互、**本页**接口 path）、**`## API 梳理`**（方法、路径、Response、字段映射）、**`## 架构设计`** / **`### 技术选型 / 约束`**、**`## 技术方案`**（文件改动清单、新增字段预估）、**`## 方案澄清事项`**（`#### Qn` + 勾选，**方案向**）、**`## 开发任务`**（层级勾选拆解）、**`## 实现计划`**（全链路数据流、跨页契约、UI、Token 等写入上述章节或计划条目，**无**独立「接口与调用顺序」等章）；亦可多轮 **`@plan.md`** 细化。
  3. **对话收尾**：若仍有开放点，Agent 应在当轮末尾**列出待用户确认项**；用户下一轮确认后再 **`Write`/`StrReplace`**，避免静默假设。

**文档不全、与已有需求冲突、边界 case →「澄清事项」须带确认（须考虑）**

Agent 在 **`/assist spec`** 落盘或迭代时，**应主动**识别下列三类缺口，并写入 **`### 澄清事项`**；每条缺口对应 **`#### Qn：…`** 下 **`- [ ]`…`⚠️`**（或已定 **`- [x]`…`✅`**），**禁止**在未经确认前把推断写进 **`plan.md`**（或兼容旧版 **`spec.md`**）的 **`## 验收标准`** 为已定 **AC**（可暂用「待 Qn 裁决」占位句）。

| 类型 | 须考虑什么 | 澄清事项里怎么写（带确认） |
| ---- | ---------- | -------------------------- |
| **当前产品文档不全** | PRD/原型**缺页、截图截断、术语歧义、无埋点/无 i18n 表、关键流程未画**；仅口述需求无书面范围 | **`#### Qn：<缺口主题>`**，下列 **`- [ ]`** 写默认假设或待补项，行末 **`⚠️`**；确认后改 **`- [x]`** + **`✅`** 并同步 **AC** |
| **与已有需求冲突** | 与本仓 **其他 `docs/features/*/spec.md`**、**同一产品域旧 PRD**、或 **线上/代码已实现行为** 不一致（入口、文案、权限、数据真源、与别 feature 的 AC 互斥） | **`#### Qn：与 … 的冲突`**，其下各 **`- [ ]`** 写「本方案 vs 现状」互斥选项 + **`⚠️`**；裁决前 **AC 不写死** |
| **边界 case 未覆盖** | 文档只写主路径；**异常、权限、空态、网络失败、返回栈、并发、降级、老数据、多语言** 等未提 | **`#### Qn：边界 …`** 下多条 **`- [ ]`** 列 case + 粗粒度期望 + **`⚠️`**；**通路**互斥时并列多行 **`- [ ]`**；明确 **Out of scope** 亦单独 **`#### Qn`** |

**与其它小节的关系**：**「结合仓库已有代码…」** 表格侧重**从代码反推事实**；本节侧重**从文档完整性与需求一致性反推待确认项**——二者都应**汇入同一套 `### 澄清事项`**，避免只在聊天里确认不落盘。

**方案选项（A / B / C…）——与用户选择**

- 当存在**多种合理需求取舍、交互形态、数据策略或发布节奏**时，Agent **应**给出 **2～4 个并列选项**（**方案 A、方案 B、方案 C**…），每项**一行摘要 + 利弊各 1 条**（可再附「推荐」标号，但**须说明理由**）。
- **回复格式约定**（对用户可见）：先列选项块，末尾固定一句：**「请回复：选 A / 选 B / 选 C，或说明组合（如 A+C）/ 自定义」**；**禁止**在未经用户选定前，把未选分支写成已定事实的 **AC** 或 **`plan.md`** 正文。
- **用户选定后的下一轮**：用户发 **`/assist spec` +「选 B」**（或自然语言等价）→ Agent **Read** 当前 **`spec.md`**（及本轮若 **`@plan.md`** 则 **`plan.md`**）→ 将**已定方案**写入 **`### 澄清事项`**（或同义 **`### 澄清项`**；记 **Qn：已选方案 B，…**）并更新 **`plan.md`**（或旧版 **`spec.md`**）**`## 验收标准`** 中与该决策相关的 **AC**；**弃选分支**可写一句「**未采纳：A/C 因 … 不纳入本 feature**」便于日后审计。
- **命名区分**：此处的 **方案 A/B/C** 指**产品/技术取舍选项**；勿与下文「**落地方式 A/B**」（脚本 vs 手写目录）混用措辞——对用户展示选项时用 **「方案 A」**，对脚本用 **「方式 A」**。

**边界 case 与「通路」选择**

- **边界 case**：异常输入、权限拒绝、网络失败、空态、并发/重入、超时、降级、与旧版兼容等；每条应能对应 **可验证** 行为（**AC-n** 或 **`### 澄清事项`** / **`### 澄清项`** 下的子 bullet）。
- **通路（分支路径）**：同一触发条件下若存在多条**互斥**用户路径（例：无权限时 **通路 1** 仅文案提示 vs **通路 2** 跳转设置 vs **通路 3** 静默重试一次），Agent **应**列出 **通路 1 / 2 / 3**（或 **P1 / P2**），各附**触发条件 + 预期 UI/数据结果**；请用户**显式选定默认通路**与**是否允许配置切换**（若产品不要求配置，则写死默认通路进 AC）。
- **落盘**：选定后，在 **`plan.md`** 的 **`## 验收标准`** 中增加或改写小节，例如 **`### 边界与通路`**：**Case-01 … 走通路 2**（须 **`@plan.md`** 或同条消息点名 **`plan.md`**；旧版 AC 仍在 **`spec.md`** 时可写回该文件直至迁移）；必要时在同一 **`plan.md`** 补**状态机 / 流程图文字描述**与接口错误码映射。

### 结合仓库已有代码列需求边界 case 与分支（推荐）

当需求会**改动或依赖**本仓已有模块时，Agent **应**在写 **`spec.md`** 前做**轻量代码对齐**（**SemanticSearch** / **Grep** / **Read** 关键入口即可，**禁止**全仓无目的扫），把**可观察事实**反映到 **`### 澄清事项`**（或 **`### 澄清项`**）与（在可写 **`plan.md`** 时）**`plan.md`** 的 **`## 验收标准`**，减少「文档与实现两张皮」。

| 动作 | 说明 |
| ---- | ---- |
| **定触点** | 从用户材料中抽出：涉及 **Screen / Activity / Route / Settings / 权限 / Deeplink / 资源 ID / API** 等关键词，映射到 **`kmp/`**、**`android/`** 下**真实路径**（与 **`android/ARCH.md`**、**`android/DEV_GUIDE.md`** 一致）。 |
| **列分支** | 记录**与现状可能冲突**的分叉：例如设置项**插入位置**（相对已有分组/项顺序）、**是否沿用**某现有图标或路由、**是否**与另一 Feature 共用入口、**前后台**谁为真源等；每条分叉对应 **Qn** 或 **方案 A/B**。 |
| **列边界 case** | 从触点推导：**权限拒绝 / 未登录 / 空列表 / 网络失败 / 重复点击 / 返回栈 / 深浅色 / 多语言 / 老版本数据** 等；能落到代码常量或错误码的**写具体名**（便于后续 **`plan.md`** 与测试对齐）。 |
| **标未知** | 代码中**不存在**或**grep 无结果**的假设，**不得**写成已定事实；写入 **Qn「待确认」** 或 **`/assist spec` 对话末**请用户确认。 |

**产出形态**：上述结果优先进入 **`### 澄清事项`**（或 **`### 澄清项`**；可交互列表见下节）；收敛后把**已定分支**同步到 **`plan.md`** 的 **`## 验收标准`**（含 **`### 边界与通路`** 若适用；须 **`@plan.md`** 或本轮可写 **`plan.md`**）。

### 澄清事项：**默认「问题 + 待确认项」= `#### Qn：` + 任务列表**

**目标形态**（与仓库内 **`docs/features/skill/spec.md`** 中 **`#### Q6：…`** + **`- [ ] … ⚠️`** 等范例一致）：读者先看到**四级标题一句点题**，其下**每条**是一个可勾选、可 diff 的**待确认命题**或**互斥选项**。

**须遵守（新建 / 首轮 PRD 归纳 / 模式 ② 写入 `spec.md` 时）**

1. **节标题**：使用 **`### 澄清事项`**（或 **`### 澄清项`** / **`### 二、澄清事项`**，三者等价）。
2. **每条澄清占一块**：**`#### Qn：<一句话主题>`**（`n` 从 **1** 起**连续递增**；主题＝**问题核心**，例：`Banner 颜色 5 种固定色板`、`往期列表是否分页`）。
3. **块内只用任务列表**：**不写**该块内的二列表「问题 | 建议」；**待确认**一律 **`- [ ]`** + 描述 + 行末 **`⚠️`**；**已定案** **`- [x]`** + 描述 + 行末 **`✅`** 或 **`✅ 已确认`**。互斥方案**并列多行** **`- [ ]`…`⚠️`**，选定后**仅**将采纳行改为 **`- [x]`…`✅`**。
4. **禁止**：澄清小节**仅有**三列表格（编号 | 问题 | 建议）而**没有任何** **`#### Qn`** 展开——**表格若保留**，仅可作节首**可选摘要**（建议 ≤5 行），且**表格每一行必须在下文有对应 `#### Qn` 任务列表**；**推荐**省略表格，**全部**用 **`#### Q1` … `#### Qn`** 写完。
5. **旧 spec 已用纯表格**：迭代时可按轮次**迁** 为 **`#### Qn`** 形态，**不**强求单次大删表（避免无关 diff 时由用户决定）。
6. **写具体**：每条 **`- [ ]`** 尽量带**可 grep 锚点**（Figma node、`Res.drawable.*`、deeplink、接口名等），避免空泛「待确认」。
7. **与方案 / 通路**：用户选定后把对应 **`- [ ]`** 改为 **`- [x]`**，并在 **`plan.md`** 的 **AC**（须 **`@plan.md`**）或本条末加**交叉引用**（如「Banner 色板见 Q6，已确认」）。

**示例（节选；新建时替换为本 feature 事实）**

```markdown
### 二、澄清事项

#### Q1：入口位置

- [x] Messages 入口：General 分组内，Permissions 之后、About 分组之前 ✅
- [ ] （备选若产品改口：放在 About 之后）未选则勿写入 AC ⚠️

#### Q2：UBT 埋点

- [x] 本期不做埋点 ✅

#### Q3：图标资源

- [x] 暂用 Permissions 同族占位图 `Res.drawable.private_b`（示例名，以仓库实际为准）✅
- [ ] 需人工干预：替换 Messages 入口正式图标资源（设计交付后更新本节并补 AC） ⚠️

#### Q4：往期列表数据加载（待确认 / 已确认 行形态示例）

- [ ] 一次性加载所有往期数据（期数有限，预计 < 50 期） ⚠️
- [ ] 分页加载 ⚠️

<!-- 用户确认「一次性 + 上限 10 期」后改为： -->

- [x] 一次性加载所有往期数据（期数有限，预计 < 10 期） ✅ 已确认
```

**Agent 注意**：行末 **✅ 已确认** / **⚠️** 为**人类扫版**约定，**勿**写入须机器解析的字段名或 CI 规则；**阶段路由与 `/assist feat`** 仍以 **`workflow-reference.md`** 章节为准，**不**依赖 emoji 判定。

**多轮用例（示意，非穷举）**

| 轮次 | 用户 | Agent |
| ---- | ---- | ----- |
| **R1** | 口述模糊需求 | 补 **Q1～Q3**，必要时给 **方案 A/B**；请用户选或答 |
| **R2** | 「选 B；Q2 选否」 | 更新 **`spec.md`** 澄清与 AC；列出剩余 **边界 case** 的 **通路 1/2** 请选 |
| **R3** | 「边界：离线用通路 1」 | **`### 边界与通路`** + **AC** 落盘；若技术细节仍多，建议 **`@plan.md`** 下一轮 |
| **R4** | **`@plan.md` + 接口草案** | 按「Assist 入口」写 **`plan.md`**，与已定 **AC/通路** 对齐 |

- **材料不足**：首轮或中间某轮信息不够时，**优先追问**（**`#### Qn`** + **`- [ ]`…`⚠️`**）或**输出方案 A/B/C** 请选，**禁止**为凑篇幅编造需求；用户逐轮补答后**累加**到 **`spec.md`**（或已解锁的 **`plan.md`**）。
- **与硬约束一致**：仅 **`/assist spec` + 材料** 且无 **`@plan.md`** / 未点名 **`plan.md`** 的每一轮，**仍只改 `spec.md`**；多轮不放宽该条。

### 「## 页面交互描述」须覆盖的范围（补充）

除 **`spec.md.tpl`** 中各小节标题外，Agent 编写或迭代 **`## 页面交互描述`** 时**应显式覆盖**（仍**不写接口**；实现选型、组件名、缓动曲线等技术细节可摘要后指向 **`plan.md`**）：

| 维度 | 须写清什么 |
| ---- | ---------- |
| **多页面串联** | **主路径与分支路径**：页 A → 页 B → …（含从子页/弹层**返回**后落在哪一屏、是否**恢复滚动位置**）；与 **deeplink 再入**、**系统返回**、**任务栈**相关的用户可见结果（用文字步骤或 **Mermaid `flowchart`/`sequenceDiagram`** 均可，**勿**写 API） |
| **涉及页面与用户路径** | 在 **`spec.md`** 仅写 **Screen / Flow / 设置分组 / 路由入口** 等**用户可见命名**与**跨页串联**（可与设计稿、**`## 澄清事项`** 交叉引用）；**勿**列 **Gradle 模块、kmp/android 路径、涉及文件矩阵**——此类写在 **`plan.md`** **`## 架构设计`**（功能模块划分）、**`## 实现计划`**、**`## 技术方案` → `### 文件改动清单`** 及 **`## 开发任务`**；若与现有壳**复用**（如共用某顶栏），在 spec **一句点明用户感知**即可，**组件表与 Token** 放 **`plan.md`** |
| **操作效果：动效与跳转** | **点击 / 切换 / 滑动手势** 之后：**全屏跳转** vs **Bottom Sheet / Dialog** vs **同页内展开**；**转场/动画**的用户感知（如「整屏收起回到卡片」「先过渡页再进消费页」——写**行为级**描述即可）；**加载/成功/失败**时的 UI 反馈（骨架、Toast、禁用态）；**不写**具体 Compose/View 属性与 duration 数值，除非产品稿已规定且需写进验收 |
| **页面结构 + 截图 / 示意结构图** | **§1「页面结构描述」**：**先**检查同目录 **`source/`** 是否已有**页面线框或截图**（见下文「推荐顺序」）；**有则嵌入**；**无或不敷用**则补 **ASCII 示意结构图**。对**纵向分区明显**的页面，除列表/表格外仍**建议**用 **①②③…** 与**分区标题或表格行**一一对应；**不替代**文字说明与 **AC** |

**§1「页面结构描述」— `source/` 截图优先，其次示意结构图（推荐顺序）**

1. **先扫描 `source/`**：在写或改 **`### 1. 页面结构描述`** 前，对 **`docs/features/<slug>/source/`**（与当前 **`spec.md`** 同目录）做 **Glob / List**；将 **`.png`、`.webp`、`.jpg`、`.jpeg`、`.gif`**（及团队约定的 **`.svg`** 线框）视为**可用页面素材**；**忽略**仅 **`README.md`** 的占位、无图文件。**`prd/`** 用于**上游 PRD / 产品文档留存**（PDF、DOCX 等），**不**作为 §1「找页面截图」的默认扫描目录；线框截图仍放 **`source/`**。**若用户 `@` 了某张路径**，以用户指定为准。
2. **有可用截图**：在 §1 **直接**使用 Markdown 嵌入：**`![分区或页面简述](./source/<文件名>)`**；图下**简短**说明分区顺序或与 **①②③** 对应关系，避免整节只有图无字。
3. **无截图、或截图不足以表达分区/条件布局**：必须按下列 **ASCII 示意结构图** 规则补全（**不以**「暂无设计稿」为由留空 §1 结构）；**单页内 ≥3 个纵向大块**、或存在**条件显示**时，**应**有示意或等价文字分区表 + 编号，便于评审与实现对齐。
4. **示意格式**：先一句 **「页面采用垂直布局，整体结构如下：」**（或等价），再接 **Markdown 代码块**（` ``` `）内 **ASCII / Unicode 框线**（`┌─┐` `├` `└` 等），块外或块内用 **① 导航栏**、**② Hint 区域** 等与 PRD/设计稿一致的编号；可辅以**同行小字**（如「仅某模式下显示」）。
5. **与表格关系**：可先 **表格**（分区 | 说明），再 **截图 / 示意**；或相反——**分区编号须一致**。
6. **边界**：截图与示意**仅表达结构与上下顺序**，不写像素、不写组件 API；**不写**接口与埋点；截图**不能**替代 **AC** 中的可测条文。

**与四 / 五小节模板的对应**：**§1** 偏单页结构（**含 `source/` 截图或 ASCII 示意**）；**§2** 偏区域内交互；**§3** 偏**单次操作**的即时反馈（含简单动效）；**§4** 偏**单页与邻接页/弹层**的进出；**§5** 偏**跨多页用户路径**的整条串联（**不写**「涉及模块」工程表；**工程涉及面**在 **`plan.md`**）——避免只在 §4 一笔带过复杂串联。
