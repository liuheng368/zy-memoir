# Feature 文档整理规范（`prd/` + `source/`）

本规范供 **`/assist spec`** 命中 **模式 ③（文件整理）** 时使用。**触发白名单**（见 **`spec-creator/SKILL.md`**「模式 ③」）：去掉 **`/assist`**、**`spec`** 与 **`@…`** 后 **trim**，剩余须**恰好**为 **`整理`** / **`整理文件`** / **`文件整理`** / **`整理一下`** 之一。在已解析的 **`docs/features/<slug>/`** 下归一 **`prd/`** 与 **`source/`**，并**同步修改** Markdown 引用路径。

**前置**：已定位 **`slug`**（用户 **`@` 目录**、**`@spec.md`** 路径，或当前 **`feat/<rest>`** 分支与 **feature** Skill 候选规则一致）。**禁止**在未确认 `slug` 时跨目录误操作。

---

## A. `prd/` 目录整理

### A.1 分析

1. **列出** `docs/features/<slug>/prd/` 下所有文件（含子目录已有内容）。
2. **主需求稿**：在 **`.md`** 中选取**一条**作为「产品需求正文」归一为 **`prd.md`**：
   - **优先**：体积最大、章节最全的 **`.md`**（常为从飞书/语雀/Notion 导出的 PRD）；
   - **排除**：`README.md`（说明档，可保留或合并后仍保留简短说明）；
   - **多条并列**且无法自动判断：在对话中列出文件名与行数，**请用户指定**哪一条改名为 `prd.md`，**禁止**擅自合并丢失信息。
3. **图片**：与主需求稿同目录、或主需求稿内引用的 **`.png` `.webp` `.jpg` `.jpeg` `.gif` `.svg`**（**不含** PDF/DOCX/PPT；二进制 PRD 保留在 `prd/` 根或子目录，见 A.3）。

### A.2 操作（须落盘）

1. 若不存在，创建 **`prd/imgs/`**。
2. 将 **主需求稿以外的**、且为 **图片扩展名** 的文件**移动**到 **`prd/imgs/`**（若已在子目录，按「与 `prd.md` 的相对引用关系」决定是否平铺到 `imgs/` 或保持子路径，以**引用最少改动**为准）。
3. 将选定的主需求稿**重命名**为 **`prd.md`**（若已叫 `prd.md` 则跳过重命名，仅整理图片与链接）。
4. **重写 `prd.md` 内引用**（及 `prd/` 下其它仍保留的 `.md` 中指向已移动图片的链接）：
   - `./foo.png` → `./imgs/foo.png`
   - `foo.png`（同目录）→ `./imgs/foo.png`
   - 已带 `./imgs/` 的**不重复**改写。
5. **Review `prd.md` 图片路径（须执行）**：**`imgs/`** 下文件**移动或重命名完成后**，对 **`prd.md`**（及上一步改过的其它 `prd/*.md`）做一次**引用核对**：
   - **列出** `prd/imgs/` 实际文件名，与文中 **`](./imgs/...)`**、**`<img src="...">`** 等路径**逐条对照**；
   - 修正：**仍指向旧子目录**、**文件名与磁盘不一致**（含空格、中文、大小写）、**错误 URL 编码**（如 `%20` 与字面空格混用导致无法打开）；
   - 对导出工具生成的**过度编码**路径：以「本地预览 / 仓库相对路径能打开」为准统一为一种写法（优先与 **`imgs/`** 内真实文件名一致）。
   - 若有链接**在 `imgs/` 中无对应文件**，在对话中**明确列出**该链接与预期文件名，**禁止**静默留断裂引用。
6. **更新** `docs/features/<slug>/spec.md` 的 **`## 关联文档`**（或等价小节）：产品原文链为 **`[PRD / 产品方案](./prd/prd.md)`**；若仍以 PDF 为主链，可并列保留 **`./prd/<文件名>.pdf`**。
7. **更新** `prd/README.md`：用 2～4 行说明「正文见 **`prd.md`**，插图见 **`imgs/`**」，避免与 `spec.md` 职责重复。

### A.3 边界

- **不删除** PDF/DOCX/PPT；仅**移动/改名**时保留可追溯路径，并在 **`prd/README.md`** 或 **`spec.md` 关联文档**中列出。
- **Git**：移动后若仓库使用 **Git LFS**，大文件仍须满足 **`.gitattributes`**；移动后执行 **`git status`** 确认 rename 被识别为 rename 而非 delete+add（必要时用 `git mv`）。

## B. `source/` 目录整理（仅 **`pages/`**，非整页图**删除**）

### B.1 保留标准（**完整页面级**）

仅当图片满足下列之一，才保留并放入 **`docs/features/<slug>/source/pages/`**：

- **整屏截图**或**单页高保真**：一屏内为完整界面（含导航/内容区），可单独支撑 **`spec.md`** §1「某一页」说明；
- **整页线框 / 单页 mock**：画布主体为一页产品结构，非局部放大块。

**不保留**（须**删除**文件，不另建子目录收纳）：图标/矢量切片、色板/色条、纹理与阴影素材、参数说明小卡、窄条参考线、角度/间距**示意小图**、与「单页完整界面」无关的装饰或 PRD 拆出的局部块。

### B.2 筛选与启发式

对 **`source/`** 根目录、`source/pages/` 下已有图片、以及历史遗留的 **`source/materials/`**（若存在）**统一过筛**：

1. **先删除目录** **`source/materials/`**（整目录；其中文件均按 B.1 视为非整页级，**不再迁移**）。
2. 对其余每张图：**符合 B.1 →** 确保位于 **`source/pages/`**；**不符合 →** **删除**（含根目录散落图、误放在 `pages/` 的非整页图）。
3. **文件名提示**（仅辅助，**以视觉语义为准**）：含 **`screen` `screenshot` `page` `wireframe` `mock` `首页` `列表` `详情` `全屏`** 等且体量明显为整页者 → 倾向保留；**极小文件**（如仅数 KB 且无整屏结构）、**`Line` `Vector` `Rectangle` `icon` `badge` `纹理` `色`** 等局部资源名 → 倾向删除。
4. **边界不清**（单张无法判断）：在对话中**贴文件名与体积**，**请用户确认**保留或删除；**未答复前**可暂保留于 `pages/` 并标注「待确认」，**禁止**为「先存着」而新建其它子目录。

### B.3 操作（须落盘）

1. 若不存在，创建 **`source/pages/`**。
2. **删除** **`source/materials/`**（若存在）。
3. 将 **`source/`** 根下（及非 `pages/` 子目录内）经 B.2 **判定为整页**的图片**移入** **`pages/`**；**非整页**的**直接删除**。
4. **更新** `spec.md`（及本 feature 下其它引用 `source/` 的 `.md`）：
   - `./source/foo.png` → `./source/pages/foo.png`（仅当 `foo.png` 被保留时）；
   - 已删除之图：**移除**对应 Markdown / HTML `<img>` 链接，或改述为文字说明，**禁止**留断裂路径。
5. **更新** `source/README.md`：写明**仅** **`pages/`** 存放整页级截图/线框，**无** `materials/`；可链 **`../prd/prd.md`**（若已存在）。

### B.4 边界

- **不移动** `prd/` 下的文件到 `source/`（两棵树独立）。
- 若 **`spec.md`** §1 使用 **HTML `<img>`** 或 **Obsidian 风格链接**，须**同样**替换或删除。

---

## C. 收尾与验证

1. **Glob / Grep**：在 **`docs/features/<slug>/`** 内搜旧路径（如 `./source/xxx.png`、**`./source/materials/`**），确保无**断裂引用**；并**二次确认** **`prd.md`** 内 **`./imgs/`** 与 **`prd/imgs/`** 实际文件一致（与 **A.2 第 5 步** 同一标准）。
2. **对话输出**：列出 **重命名 / 移动 / 删除** 清单（旧路径 → 新路径 或 **已删除**），便于 **MR** 审查。
3. **与 spec-creator 其它模式**：整理完成后，若用户继续 **`/assist spec` + 需求材料**，按 **模式 ②** 正常写 **`spec.md`**；**AC** 仍默认在 **`plan.md`**（须 **`@plan.md`**）。

---

## D. 与模板默认值的关系

- **`create-feature-spec.mjs`** 新建目录时仍为 **`prd/README.md`** + **`source/README.md`** 根占位；**整理**是在 feature 已存在杂乱导出文件后的**可选归一**，不改变脚本默认首帧。
- 整理完成后，可将 **`templates/prd/README.md`** / **`templates/source/README.md`** 的推荐结构**同步**到本 feature 的 README（不必回写模板，除非团队统一改默认骨架）。
